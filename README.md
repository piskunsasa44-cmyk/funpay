# BR Price Monitor v2 — мониторинг цен виртов Black Russia на FunPay

Отдельный процесс, не трогает основной бот на Railway.

## Что изменилось в v2 (если у тебя был v1)

Причина, по которой бот слал "случайные" лоты, а не реально дешёвые — почти
наверняка отсутствие авторизации. Без `golden_key` страница FunPay может
отдавать цены не в рублях, и тогда любое маленькое число проходило порог
"<=26₽". Теперь:

- `FUNPAY_GOLDEN_KEY` — **обязательная** переменная. Без неё скрипт не запустится
  и сразу напишет в лог понятную причину
- Добавлена проверка: если медианная цена по всем найденным лотам подозрительно
  низкая (< 5₽) — скрипт НЕ шлёт алерты в этом цикле и пишет ошибку в лог,
  вместо того чтобы спамить мусором
- Есть режим `DEBUG_ROWS` — печатает в лог сырой текст строк таблицы, если
  что-то всё равно парсится не так

## Переменные окружения

| Переменная | Обязательна | По умолчанию | Что это |
|---|---|---|---|
| `TELEGRAM_BOT_TOKEN` | да | — | Токен бота от @BotFather |
| `TELEGRAM_CHAT_ID` | да | — | Твой chat id (узнавал через getUpdates) |
| `FUNPAY_GOLDEN_KEY` | **да** | — | Cookie твоей сессии FunPay, см. ниже |
| `PRICE_THRESHOLD_RUB` | нет | `26` | Порог цены за 1kk в рублях |
| `POLL_INTERVAL_SECONDS` | нет | `180` | Частота опроса FunPay |
| `SANITY_MIN_PRICE_RUB` | нет | `5` | Ниже этой медианной цены — считаем, что парсинг сломан |
| `DEBUG_ROWS` | нет | `0` | Если >0 — печатает в лог сырые данные первых N строк таблицы |
| `REALERT_COOLDOWN_SECONDS` | нет | `3600` | Не слать повтор по тому же лоту чаще раза в час, если цена не упала ещё ниже |

### Как достать `FUNPAY_GOLDEN_KEY`

1. Зайди на funpay.com в браузере под своим аккаунтом (тем, с которого торгуешь)
2. F12 → Application (Chrome) / Storage (Firefox) → Cookies → funpay.com
3. Найди cookie `golden_key`, скопируй значение целиком
4. Никому не показывай это значение — это ключ доступа к твоей сессии на FunPay,
   по нему можно действовать от твоего имени на сайте

## Деплой на Railway

1. В сервисе → **Variables** — впиши все переменные из таблицы выше
   (минимум обязательные три: токен, chat id, golden_key)
2. **Settings → Deploy → Custom Start Command**: `python monitor.py`
3. Redeploy
4. Открой **Deploy Logs**, должно быть:
```
Старт мониторинга v2. Порог=26₽/1kk, серверы=[1, ..., 30], интервал=180с, golden_key=задан
Распарсено лотов: N
Лотов ниже порога 26₽: M
```
5. При первом успешном запуске бот сам пришлёт тебе в Telegram сообщение
   "BR Price Monitor запущен и работает" — это подтверждение, что всё на связи

## Если снова видишь странности (лотов слишком много / слишком мало)

1. В Variables поставь `DEBUG_ROWS=5`, Redeploy
2. В Deploy Logs найдёшь блоки вида:
```
RAW ROW #0: №05 Лагуна | Описание | Продавец | 500 | 26.5
  .tc-server -> '№05 Лагуна'
  .tc-amount -> '500'
  .tc-price -> '26.5'
```
3. Сравни эти числа с тем, что видишь на самой странице FunPay глазами в браузере
4. Скинь мне эти строки из лога — по ним я точно увижу, что и где парсится не так,
   и поправлю селекторы прицельно, а не угадывая

## Если "Медианная цена ... — это нереально дёшево" в логах

Значит `golden_key` не подцепился (опечатка в названии переменной в Railway,
либо cookie устарел/протух — зайди на funpay.com заново в браузере и скопируй
свежий). Алерты в этом случае не шлются — это специально, чтобы не закидывать
тебя мусором, пока проблема не починена.

## Важно

- Не ставь `POLL_INTERVAL_SECONDS` меньше 60 — частые запросы могут привести
  к временной блокировке по IP на стороне FunPay
- Это парсинг публичной страницы лотов под своей сессией, без обхода каких-либо
  технических ограничений доступа — но держи частоту запросов разумной
